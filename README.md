# retrime-red

Архитектура бэкенда (предложение и минимальная реализация)
- Проблема токенов в URL: токен, переданный в фрагменте URL после #token=..., не отправляется на сервер браузером. Это значит, что любой токен в hash части не может быть автоматически прочитан серверной логикой. Рекомендовано использовать OAuth2/OIDC через Authorization Code Flow с коротким временем жизни сессионного токена, который хранится в HttpOnly куках.
- Предлагаемая архитектура:
  - Edge Gateway (Cloudflare Workers): авторизация и API. Обрабатывает вход, хранение сессий и проксирование к API жалоб.
  - IdP / Auth Provider: выдает коды авторизации и токены (id_token/access_token). В продакшне — внешний поставщик (например Okta, Auth0, Keycloak).
  - KV/Storage: хранение жалоб и сессий.
  - Frontend (Cloudflare Pages): статические страницы, которые после аутентификации получают данные через защищённый API и скрывают кнопку входа при наличии активной сессии.
  - API жалоб: /api/complaints, /api/complaints/{id} и т.д. Доступ к API — только для аутентифицированных.
- Реализация (минимальная): edge-auth (Cloudflare Worker) реализует:
  - Поток входа через /login и /auth/callback (Mock IdP внутри воркера на этапе разработки, далее замена на реальный IdP).
  - API жалоб с сохранением в KV: создание, список, детализация.
  - /dashboard как базовая тестовая страница для проверки аутентификации.
- Безопасность: HttpOnly куки sess_id, валидация сессии через KV, права доступа к данным жалоб. В продакшне — JWT в IdP, JWKS валидация, CSRF и rate-limiting.
- Как начать:
  1) Развернуть edge-auth на Cloudflare Workers (настроить wrangler.toml с реальными IDs KV и account_id).
  2) Подключить KV для COMPLAINTS_KV и настроить миграцию/хранение жалоб.
  3) В интеграции с фронтендом — использовать /api/complaints и скрывать вход, если есть активная сессия.
